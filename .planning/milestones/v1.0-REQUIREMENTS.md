# Requirements — v1.0 Stabilization

**Milestone:** v1.0 Stabilization
**Goal:** Restore a clean TypeScript build, close three unauthenticated mutation endpoints, and fix session ownership loss on restart.
**Status:** Active

---

## v1 Requirements

### Build

- [x] **BUILD-01**: Developer can run `tsc --noEmit` to a clean exit (errors in upload.ts, RouterAgent.ts, councilDispatch.ts all resolved)
- [x] **BUILD-02**: Upload rate limiter is functional (`createRateLimitMiddleware` import restored in upload.ts)
- [x] **BUILD-03**: Abort timeout is correctly derived from `timeoutMs` in `RouterAgent.sampleAgents()` (no bare out-of-scope reference; positional argument mismatch at councilDispatch.ts:103 removed)

### Session

- [x] **SESSION-01**: Session `owner_api_key_hash` is persisted to SQLite on session creation (migration 008 applied; `SessionStore.createSession()` accepts and stores hash)
- [x] **SESSION-02**: Session ownership check reads from SQLite, not in-memory registry, and survives process restarts without 403 regression

### Auth

- [x] **AUTH-01**: POST /dispatch rejects requests where caller's hashed API key does not match `session.owner_api_key_hash`
- [x] **AUTH-02**: PATCH /session/:id/revert requires session ownership (requireOwnedSession backed by SQLite)
- [x] **AUTH-03**: POST /config requires API key authentication and rejects invalid `default_mode` values (only `solo` and `council` accepted)
- [x] **AUTH-04**: GET /config and GET /quota require API key authentication

## Out of Scope

- New features of any kind — this milestone is stabilization only
- bcrypt/argon2 for API key hashing — incompatible with synchronous node:sqlite; SHA-256 sufficient for high-entropy keys
- OpenRouter key validation on every request — latency and quota cost
- Multi-user isolation, JWT, OAuth, or session cookies
- Phase 4 (Structured Workbench) and Phase 5 (Full GoA+MoA) — post-stabilization
