# Roadmap: FreeCouncil v1.0 Stabilization

## Overview

Three sequential phases restore a safe, compiling codebase. Phase 1 fixes the TypeScript build so that `tsc --noEmit` exits cleanly and the test suite can run. Phase 2 persists session ownership to SQLite so that process restarts cannot invalidate existing sessions. Phase 3 wires authentication middleware against the now-persistent ownership data, closing all five unauthenticated or under-authenticated routes. Each phase is a hard prerequisite for the next: the compiler must be green before schema changes can be verified, and the schema must exist before auth guards are correct.

## Phase Details

### Phase 1: TypeScript Build

**Goal**: The TypeScript compiler reports zero errors on the entire codebase
**Depends on**: Nothing (first phase)
**Requirements**: BUILD-01, BUILD-02, BUILD-03
**Success Criteria** (what must be TRUE):

  1. `tsc --noEmit` exits with code 0 and prints no diagnostics
  2. The `createRateLimitMiddleware` symbol is imported and applied in `upload.ts` so the upload rate limiter is active at runtime
  3. `RouterAgent.sampleAgents()` derives the AbortSignal from `AbortSignal.timeout(timeoutMs)` with no reference to an out-of-scope `abortSignal` variable
  4. `councilDispatch.ts` calls `sampleAgents()` without a trailing AbortSignal argument, so `timeoutMs` receives a number and not an AbortSignal object

**Plans**: 2

### Phase 2: Session Ownership Persistence

**Goal**: Session ownership survives process restarts because `owner_api_key_hash` is durable in SQLite
**Depends on**: Phase 1
**Requirements**: SESSION-01, SESSION-02
**Success Criteria** (what must be TRUE):

  1. After a new session is created, the `sessions` SQLite table row contains a non-null `owner_api_key_hash` value
  2. After a process restart, a request to an ownership-gated route with the correct API key succeeds (no spurious 403)
  3. `requireOwnedSession` reads ownership from `SessionStore` (SQLite) and never from the in-memory `SessionRegistry`
  4. An existing session row with a NULL `owner_api_key_hash` (pre-migration) is treated as unowned, not as a crash

**Plans**: TBD

### Phase 3: Auth Middleware Wiring

**Goal**: Every mutation and configuration route requires authentication; unauthorized callers receive 401 or 403
**Depends on**: Phase 2
**Requirements**: AUTH-01, AUTH-02, AUTH-03, AUTH-04
**Success Criteria** (what must be TRUE):

  1. A request to POST /dispatch with no `Authorization` header is rejected before any quota is consumed
  2. A request to POST /dispatch with a valid API key but a `sessionId` owned by a different key returns 403
  3. A request to PATCH /session/:id/revert from a non-owner returns 403
  4. A request to POST /config with no API key is rejected; a request with a valid key but `default_mode` set to any value other than `solo` or `council` returns 400
  5. Requests to GET /config and GET /quota with no API key are rejected
